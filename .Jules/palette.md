## 2024-05-19 - Destructive Actions and Hover States
**Learning:** Adding a destructive action (like a delete button) to a list item improves UX significantly but requires careful balancing. Hiding the button behind a hover/focus state keeps the UI clean, while native confirmation dialogues and loading spinners provide critical safety against accidental clicks and double submissions. Additionally, making sure the loading state remains visible (by overriding the hover opacity when active) prevents the button from disappearing mid-operation if the user moves their mouse.
**Action:** When adding inline destructive actions to list items, always use a combination of hover/focus reveal, `aria-label`, native confirmation (e.g., `window.confirm`), and a loading state that forces the element to remain visible during the async operation.
## 2024-04-24 - Structured Empty States and Loading Indicators
**Learning:** For interfaces with asynchronous data loading (like a search page), providing a continuous visual feedback loop dramatically improves UX. Showing a clear loading spinner during requests prevents user confusion about system status. When the request resolves with no data, displaying a structured empty state—using a dashed border, a prominent yet subdued icon, and clear instructional text—is far superior to returning a blank page or a raw text string, as it keeps the user oriented and guides their next action.
**Action:** When implementing pages or components that fetch data, always pair an `isLoading` state (e.g., `<Loader2 className="animate-spin" />`) with a robust empty state pattern (icon, heading, subtext inside a styled container) for when the query returns an empty array.
## 2024-05-20 - Keyboard Shortcuts and Visual Hints for Search

## 2024-05-23 - Disconnected Labels on Custom UI Trigger Components
**Learning:** Custom UI components that use an underlying native element as a trigger (like a `<Button>` triggering a `<Popover>`) often forget to pass down standard HTML attributes like `id`. This breaks the connection between explicit `<label htmlFor="...">` and the interactive trigger, making it impossible for screen reader users to understand what the button controls, and preventing users from clicking the label to focus the trigger. Adding `aria-hidden="true"` to purely decorative icons inside these triggers further reduces unnecessary noise.
**Action:** Always verify that custom trigger components (like Date Pickers, Selects, or Dialogs) correctly accept and forward the `id` prop to their underlying focusable element (e.g., `<Button id={id}>`) so that form labels can be properly associated.

## 2026-05-12 - Missing Radix UI Dialog and Sheet Descriptions

## 2026-05-10 - Undefined Aria-DescribedBy for UI Dialog/Sheet Contents
**Learning:** When using Radix UI (or shadcn/ui) `DialogContent` or `SheetContent` components, if a visible or structurally necessary description isn't needed or present (no `DialogDescription` or `SheetDescription` rendered), screen readers can sometimes complain about a missing description, and Radix will output a missing description console warning.
**Action:** When a Dialog or Sheet intentionally lacks a description, explicitly pass `aria-describedby={undefined}` to the content component (e.g., `<DialogContent aria-describedby={undefined}>`). This explicitly satisfies accessibility requirements and suppresses warnings without needing to clutter the DOM with visually hidden text.
## 2026-05-06 - Fixing Radix UI Dialog and Sheet Accessibility Descriptions
**Learning:** To resolve missing description accessibility warnings for Radix UI (or shadcn/ui) `Dialog` or `Sheet` components, explicitly passing `aria-describedby={undefined}` is the official way to opt-out ONLY if no supplementary description is intended. However, if a visually hidden `<DialogDescription className="sr-only">` (or `<SheetDescription className="sr-only">`) is actually present in the component, you must NOT pass `aria-describedby={undefined}` to the wrapper content component. Doing so overrides and breaks the auto-generated internal IDs that Radix UI uses to link the component to the screen reader description.
**Action:** Always check if a `<DialogDescription className="sr-only">` or `<SheetDescription className="sr-only">` is being rendered inside a `DialogContent` or `SheetContent` before blindly adding `aria-describedby={undefined}` to suppress linter warnings.
